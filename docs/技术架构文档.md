# **技术架构文档 (Technical Architecture Document)**

**项目名称:** VentureAgent 创新创业多智能体教学系统

**文档版本:** V1.0

**设计原则:** 高稳定性、长期支持 (LTS)、低运维风险、社区生态丰富

## **1\. 架构设计理念 (Design Philosophy)**

鉴于教育系统对可用性和数据一致性的极高要求，本系统的技术选型严格遵循\*\*“保守创新”\*\*原则：

* **核心业务前沿化**：在业务逻辑层保留最先进的 LangGraph 多智能体协同和超图校验引擎。  
* **底层基建传统化**：在数据存储、中间件、Web 框架上，全部采用发布 5 年以上、有大规模企业级生产落地案例、社区生态极度丰富的 LTS（长期支持）版本技术，最大程度避免“踩坑”和框架底层 Bug。

## **3\. 总体系统架构图 (System Architecture)**

系统整体划分为五大逻辑层，采用严格的单向依赖设计：

\[ 前端展示层 (UI) \]  \--\> React, Ant Design, ECharts (数据可视化)  
        | (HTTP/RESTful API \+ WebSocket)  
\[ 接入与网关层 \]     \--\> Nginx (反向代理/负载均衡) \-\> FastAPI (应用接口)  
        |  
\[ 智能体编排层 \]     \--\> LangGraph (Router, Coach, Critic, Evaluator 等五大 Agent)  
        | (State Checkpointing)  
\[ 核心业务逻辑层 \]   \--\> HG Constraints Engine (H1-H15规则), 图谱检索 (GraphRAG), 文档解析  
        | (Cypher, SQL, Redis API)  
\[ 数据与存储底座 \]   \--\> Neo4j (知识图谱/超图), PostgreSQL (用户/向量/状态), Redis (缓存/队列)

## **4\. 核心模块详细设计 (Module Detailed Design)**

### **4.1 数据流与状态持久化设计 (State Management)**

多智能体对话的难点在于“记忆”的丢失和错乱。

* **技术方案**：使用 LangGraph 官方支持的 langgraph-checkpoint-postgres。  
* **实现逻辑**：每一次 Agent 的状态流转（例如从“问题诊断”进入“痛点追问”），其全局 Context（包含学生输入、提取的节点、图谱检索结果）都会被序列化存入 PostgreSQL 中。如果程序意外重启，可以精确地从断点恢复，保证辅导过程的绝对连续性。

### **4.2 超图 (Hypergraph) 在 Neo4j 中的稳定建模方案**

Neo4j 是属性图（Property Graph），原生不支持一条边连接多个节点。为了以最稳定的方式实现需求中的超图（如 Value\_Loop\_Edge），采用**超边节点化 (Hyperedge-as-a-Node)** 设计模式：

* **数据结构**：创建一个类型为 Hyperedge 且 Label 为 Value\_Loop 的实体节点。  
* **连接方式**：将相关的 Technology、Market、Project 节点通过标准二元关系（如 PART\_OF）指向该超边节点。  
* **优势**：完美复用 Neo4j 极其成熟的 Cypher 查询语言和图算法，规避了实验性超图数据库的底层 Bug。

### **4.3 异步任务编排设计 (Async Task Orchestration)**

针对教师端的自动化批改（形成性评价 Agent），因涉及数十万字的文本阅读和交叉验证，耗时可能长达数分钟。

* **技术方案**：请求到达 FastAPI 后，立即返回 Task\_ID。后台由 **Celery Workder** 接管任务。  
* **执行流程**：  
  1. Celery 调度 parse\_docs.py 进行文档切块。  
  2. 调用 LLM 结合 Neo4j 进行证据链对比。  
  3. 任务完成后，将批改报告（Rubric Table \+ Evidence Trace）写入 PostgreSQL。  
  4. 前端通过轮询或 WebSocket 获取结果，彻底杜绝 Nginx 504 Gateway Timeout 报错。

### **4.4 逻辑约束与拦截引擎 (Constraints Engine)**

将 PRD 中的 H1-H15 规则转化为确定性的 Python 代码，作为 Agent 执行前后的“安全护栏”。

* **实现逻辑**：采用经典的策略模式（Strategy Pattern）。  
  \# 稳定可靠的代码组织方式示例  
  class ConsistencyChecker:  
      def \_\_init\_\_(self, neo4j\_driver):  
          self.db \= neo4j\_driver  
          self.rules \= \[CheckH1Mismatch(), CheckH8UnitEconomics(), ...\] \# 注册H1-H15规则

      def run\_audit(self, extracted\_nodes):  
          \# 遍历执行所有硬编码规则，不依赖LLM的幻觉  
          for rule in self.rules:  
              result \= rule.evaluate(self.db, extracted\_nodes)  
              if result.status \== 'FAIL':  
                  return result \# 立即熔断并返回给 Critic Agent 进行追问  
          return PASS

## **5\. 安全与运维设计 (Security & DevOps)**

### **5.1 数据隔离与隐私安全**

* 采用 **多租户架构 (Multi-tenancy)**，在 PostgreSQL 和 Neo4j 中通过 school\_id 和 class\_id 实施严格的行级数据权限隔离。  
* 学生的原始创意文件（BP、路演 PPT）在传入 LLM 前，必须经过本地化脱敏脚本（正则匹配剔除身份证、学号等敏感信息）。

### **5.2 监控与告警体系 (Monitoring)**

采用传统的运维老三样，确保系统黑盒变白盒：

* **Prometheus \+ Grafana**: 监控 FastAPI 的 QPS、响应时间，以及 PostgreSQL/Neo4j 的连接池状态。  
* **LangSmith / Phoenix**: （可选，成熟的 LLM 运维平台）专用于追踪 Agent 的内部调用链（Trace），便于开发人员排查为何 Agent 触发了错误的 H1 规则。  
* **Sentry**: 捕获所有 Python 代码级别的 Exception，实时发送钉钉/企微告警。

## **6\. 总结 (Conclusion)**

本技术架构**坚决抵制为了炫技而引入不成熟组件的诱惑**。

通过 Python/FastAPI \+ LangGraph \+ Neo4j \+ PostgreSQL 这套经历了无数中大型企业实战检验的技术栈，我们为 VentureAgent 打造了一个坚如磐石的底座。这套架构不仅能完美支撑复杂的“超图逻辑推演”和“多智能体协作”，还能保证在未来 3-5 年内，系统面临高校高并发教学任务时，依然保持极低的故障率和极佳的可维护性。
